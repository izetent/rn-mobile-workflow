# Fix Selection

Use this after the root cause is known.

## Direct fix

Apply the fix directly when:

- the root cause is clear
- the change boundary is small
- there is no meaningful architecture tradeoff

## Present options first

Present options before editing when:

- multiple fixes are viable
- the fix changes ownership boundaries
- the fix affects API shape
- the fix trades performance against flexibility
- the fix increases maintenance cost

## Option format

For each option, state:

- what changes
- why it works
- tradeoffs
- risk
- recommended choice
